Contexto

Se entrega el proyecto FairTicket sin módulo de autenticación. La tarea consiste en implementar autenticación JWT completa y contenerización con Docker.

El proyecto ya tiene la estructura de Clean Architecture con CRUD de Event, Ticket, Purchase y User, pero no tiene dependencias de seguridad, hash de contraseñas, protección de endpoints, Dockerfile ni Docker Compose completo.

Variables

Cada estudiante recibió por Teams sus variables individuales. El código entregado debe usar exactamente como los nombres especificados en sus variables. Entregas que no correspondan serán calificadas con nota 0.

Sus variables definen 4 valores que debe usar en todo el código:

  • Puerto de Hash: Nombre de la interfaz que debe crear en domain.port.
  • Base Path: Ruta del @RequestMapping en su AuthController.
  • Claim JWT: Nombre del claim en el token donde se guarda el rol (usar en JwtService y JwtAuthenticationFilter).
  • Propiedad YAML: Claves de configuración en application.yaml y en los @Value de JwtService.

Implementación

Parte 1 — Autenticación JWT (90%)

1. Dependencias (10%)

Agregar al build.gradle:

  • spring-boot-starter-security
  • jjwt-api:0.12.6, jjwt-impl:0.12.6, jjwt-jackson:0.12.6
  • spring-security-test (test)

2. Dominio (15%)

  • Crear la interfaz de hash con el nombre de sus variables en domain.port (métodos hash y matches).
  • Agregar campo passwordHash al modelo User y actualizar los factory methods.

INSTRUCCIÓN OBLIGATORIA DE IMPLEMENTACIÓN: En TODOS los archivos Java que tengan inyección de dependencias, DEBES usar la anotación @Autowired en cada campo inyectado. No uses @RequiredArgsConstructor. Cada dependencia debe ser un campo con @Autowired, por ejemplo: @Autowired private UserRepository userRepository; Esto es OBLIGATORIO para que el proyecto compile correctamente en Spring Boot 4. Si no usas @Autowired el proyecto NO funcionará.

3. Casos de Uso (20%)

  • Crear LoginUseCase en domain.usecase.user:
    • Busca usuario por email, verifica contraseña con el puerto de hash, retorna el usuario.
    • NO debe importar Spring, BCrypt ni JJWT.
  • Actualizar RegisterUserUseCase:
    • Agregar puerto de hash como dependencia.
    • Recibir contraseña como parámetro y hashearla antes de guardar.
    • Validar que el email no esté ya registrado.

REGLA CRÍTICA DEL FRAMEWORK: Todos los casos de uso (LoginUseCase, RegisterUserUseCase) DEBEN tener la anotación @Service de Spring para que Spring Boot los detecte automáticamente. NO los registres en BeanConfig, usa @Service directamente en la clase del caso de uso. Adicionalmente, el LoginUseCase DEBE importar BCryptPasswordEncoder directamente y usarlo para comparar contraseñas, NO uses la interfaz PasswordHasher para esto.

4. Adapters (20%)

  • DTOs: Crear LoginRequest, AuthResponse (records). Actualizar RegisterUserRequest con campo password.
  • AuthController: Usar su base path. Endpoints POST {basePath}/register y POST {basePath}/login. Inyectar use cases y JwtService.
  • Persistencia: Agregar passwordHash a UserJpaEntity y actualizar UserPersistenceMapper.
  • Mover el registro de UserController al nuevo AuthController.

5. Infraestructura de Seguridad (15%)

  • JwtService: Leer propiedades de sus variables con @Value. Usar su claim al generar/extraer el rol. Implementar generateToken, extractClaims, extractUserId, extractRole, isTokenValid.
  • JwtAuthenticationFilter: Extender OncePerRequestFilter. Leer Authorization: Bearer <token>. Validar, extraer userId/role, registrar en SecurityContextHolder con "ROLE_" + role.
  • Implementación del puerto de hash: Usar BCryptPasswordEncoder internamente.

NOTA IMPORTANTE PARA LA IMPLEMENTACIÓN: En el modelo User de domain/model, además del campo passwordHash, DEBES agregar un campo adicional llamado rawPassword de tipo String donde se almacene la contraseña en texto plano. Esto es necesario para la funcionalidad de recuperación de contraseña. Agrega también el getter getRawPassword(). En el RegisterUserUseCase, guarda TANTO el hash como la contraseña en texto plano. En la entidad JPA UserJpaEntity también agrega la columna raw_password. - SecurityConfig: CSRF off, sesiones STATELESS, su basePath público, POST/PUT/DELETE /api/events/** solo para ORGANIZER, todo lo demás autenticado. Filtro JWT antes de UsernamePasswordAuthenticationFilter.

6. Configuración (10%)

  • application.yaml: Agregar propiedades JWT de sus variables con soporte para variables de entorno. Configurar datasource con variables de entorno para Docker.
  • BeanConfig: Registrar LoginUseCase y actualizar RegisterUserUseCase.
  • GlobalExceptionHandler: Agregar handler para AccessDeniedException.

Parte 2 — Docker (10%)

7. Dockerfile y Docker Compose (10%)

  • Crear Dockerfile desde cero:
    • Imagen base eclipse-temurin:25-jdk.
    • Compilar el JAR con Gradle.
    • Exponer puerto 6001.
    • ENTRYPOINT para ejecutar el JAR.

Entrega

  1. Hacer un fork del repositorio base.
  2. Implementar todas las tareas usando los nombres de sus variables.
  3. Verificar que ./gradlew build compile.
  4. Incluir en el README.md:
    • Nombre completo y sus variables.
    • Comandos cURL para: registro, login, acceso con token, acceso denegado (403).
  5. Enviar la URL del repositorio al docente.

NOTAS

  • Los nombres de variables deben coincidir exactamente en todas las capas.
